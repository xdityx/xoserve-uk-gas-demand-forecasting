# UK NTS Gas Demand Forecasting

[![CI](https://github.com/xdityx/xoserve-uk-gas-demand-forecasting/actions/workflows/ci.yml/badge.svg)](https://github.com/xdityx/xoserve-uk-gas-demand-forecasting/actions/workflows/ci.yml)

A production-oriented daily UK National Transmission System gas-demand forecasting project using finalized National Gas demand, Met Office HadCET temperature, recursive backtesting, and a freshness-aware FastAPI service.

## Current status

The dates below describe the checked-in 2026-07-13 snapshot; `/health` is the runtime source of truth after scheduled refreshes.

- Demand data: 2,004 daily observations from 2021-01-11 through 2026-07-07.
- Weather data: HadCET daily mean temperature through 2026-07-12.
- Source freshness: the API reports source age and rejects forecasts when demand is more than 14 days old, unless `allow_stale=true` is explicitly supplied.
- Models served: ARIMA, SARIMA, weather-aware linear regression, and weather-aware RandomForest.
- Forecast output: dated point forecasts, lower/upper intervals, forecast origin, model metadata, and freshness metadata.

Deployment target: [Render API](https://xoserve-uk-gas-demand-forecasting-1.onrender.com) and [interactive docs](https://xoserve-uk-gas-demand-forecasting-1.onrender.com/docs).

## Honest out-of-sample results

The authoritative results are generated by `python scripts/validate_oos.py` and stored in [reports/oos_results.json](reports/oos_results.json). The table below is the 2026-07-07 data snapshot; the JSON report is the runtime source of truth after scheduled refreshes. Evaluation uses five expanding rolling-origin folds with a fixed 14-day horizon. Lag features are generated recursively, so holdout demand is never used as a future input.

| Model | Mean MAE (GWh) | Mean RMSE (GWh) |
|---|---:|---:|
| Persistence | 146.62 | 172.12 |
| Linear regression + HDD | 192.68 | 227.14 |
| RandomForest + HDD | 155.64 | 183.01 |
| ARIMA | 148.13 | 173.57 |
| SARIMA | **128.78** | **157.88** |

These results cover holdouts from 2026-04-29 through 2026-07-07. SARIMA is the strongest tested model in this window. The weather-aware models use realized holdout temperatures, so their scores exclude weather-forecast error and should be treated as an upper-bound diagnostic. They did not outperform SARIMA in this backtest.

The earlier in-sample MAEs are intentionally no longer presented as forecast evidence.

## Leakage-safe features

- Heating Degree Days: `max(15.5 - mean_temp, 0)`
- Demand lag 1
- Demand lag 7
- Previous-seven-day rolling mean

The rolling feature is `demand.shift(1).rolling(7).mean()`. Shifting before rolling prevents the current target from entering its own feature row.

Weather-aware multi-day forecasts are recursive: each predicted demand is appended to history before constructing the next day's lag and rolling inputs.

## Data sources and refresh

- Demand: [National Gas Gas Data Portal](https://www.nationalgas.com/our-businesses/operational-data/gas-data-portal), publication `PUBOB652` ("Demand Actual, NTS, D+6"), mscm converted to GWh with the documented factor 11.078.
- Weather: [Met Office HadCET daily mean temperature](https://hadleyserver.metoffice.gov.uk/hadobs/hadcet/data/download.html).

Refresh both sources and regenerate validation:

```bash
python scripts/update_data.py
python scripts/validate_oos.py
```

The scheduled GitHub workflow performs the same refresh and commits changed source data and validation output. D+6 demand is expected to lag the current date by approximately six days.

## Getting started

Requirements: Python 3.12 and pip.

```bash
pip install -r requirements.txt
python -m pytest tests/ -v
uvicorn api.app:app --reload
```

Run the API container:

```bash
docker build -t gas-forecast-api .
docker run --rm -p 8000:8000 gas-forecast-api
```

The Docker image serves FastAPI on port 8000 and includes the source data and generated validation report.

## API

Health and freshness:

```bash
curl http://127.0.0.1:8000/health
```

SARIMA forecast:

```bash
curl -X POST http://127.0.0.1:8000/forecast \
  -H "Content-Type: application/json" \
  -d '{"days": 7, "model_type": "sarima"}'
```

Weather-aware recursive forecast. One forecast temperature is required per day:

```bash
curl -X POST http://127.0.0.1:8000/forecast \
  -H "Content-Type: application/json" \
  -d '{"days": 3, "model_type": "linear", "mean_temperatures": [7.1, 6.8, 6.5]}'
```

Endpoints:

- `GET /health`: service and demand-source freshness.
- `POST /forecast`: dated forecasts with uncertainty bounds and metadata.
- `GET /compare`: the persisted, reproducible rolling-origin validation report.
- `GET /docs`: interactive OpenAPI documentation.

Supported `model_type` values are `arima`, `sarima`, `linear`, and `random_forest`. Forecast horizons are capped at 31 days.

## Limitations

- CET is a national proxy rather than a demand-weighted spatial weather measure.
- Weather-aware validation uses realized temperature; operational accuracy depends on supplied weather forecasts.
- Calendar, holiday, price, storage, and structural-demand effects are not yet modeled.
- Forecast intervals are model-based for ARIMA/SARIMA and empirical training-residual intervals for the regression models; they are not yet formally calibrated.
- The fixed mscm-to-GWh factor is an analytical approximation rather than a daily calorific-value conversion.
