# UK NTS Gas Demand Forecasting

[![CI](https://github.com/xdityx/xoserve-uk-gas-demand-forecasting/actions/workflows/ci.yml/badge.svg)](https://github.com/xdityx/xoserve-uk-gas-demand-forecasting/actions/workflows/ci.yml)

A production-oriented daily UK National Transmission System gas-demand forecasting project using provisional and finalized National Gas demand, Met Office HadCET temperature, frozen-forecast scoring, recursive backtesting, and a freshness-aware FastAPI service.

## Current status

The dates below describe the checked-in 2026-07-14 snapshot; `/health` is the runtime source of truth after scheduled refreshes.

- Finalized demand (`D+6`): authoritative history through 2026-07-08.
- Provisional demand (`D+1`): operational tail through 2026-07-13. A D+6 value always replaces D+1 on overlap.
- Weather data: HadCET daily mean temperature through 2026-07-13.
- Source freshness: the API reports each demand vintage and rejects forecasts when operational demand is more than 14 days old, unless `allow_stale=true` is explicitly supplied.
- Models served: ARIMA, SARIMA, weather-aware linear regression, and weather-aware RandomForest.
- Scheduled output: immutable, dated SARIMA forecasts with intervals and persistence/weekly-naive benchmarks.

Deployment target: [Render API](https://xoserve-uk-gas-demand-forecasting-1.onrender.com) and [interactive docs](https://xoserve-uk-gas-demand-forecasting-1.onrender.com/docs).

## Honest out-of-sample results

The authoritative results are generated by `python scripts/validate_oos.py` and stored in [reports/oos_results.json](reports/oos_results.json). The table below is the 2026-07-08 data snapshot; the JSON report is the runtime source of truth after scheduled refreshes. Evaluation uses five expanding rolling-origin folds with a fixed 14-day horizon. Lag features are generated recursively, so holdout demand is never used as a future input.

| Model | Mean MAE (GWh) | Mean RMSE (GWh) |
|---|---:|---:|
| Persistence | 181.30 | 214.00 |
| Linear regression + HDD | 204.98 | 237.14 |
| RandomForest + HDD | 198.32 | 222.74 |
| ARIMA | 185.91 | 218.83 |
| SARIMA | **156.76** | **189.61** |

These results cover holdouts from 2026-04-30 through 2026-07-08. SARIMA is the strongest tested model in this window. The weather-aware models use realized holdout temperatures, so their scores exclude weather-forecast error and should be treated as an upper-bound diagnostic. They did not outperform SARIMA in this backtest.

The earlier in-sample MAEs are intentionally no longer presented as forecast evidence.

## Leakage-safe features

- Heating Degree Days: `max(15.5 - mean_temp, 0)`
- Demand lag 1
- Demand lag 7
- Previous-seven-day rolling mean

The rolling feature is `demand.shift(1).rolling(7).mean()`. Shifting before rolling prevents the current target from entering its own feature row.

Weather-aware multi-day forecasts are recursive: each predicted demand is appended to history before constructing the next day's lag and rolling inputs.

## Data sources and refresh

- Provisional demand: [National Gas Gas Data Portal](https://www.nationalgas.com/our-businesses/operational-data/gas-data-portal), publication `PUBOB637` ("Demand Actual, NTS, D+1").
- Finalized demand: the same portal, publication `PUBOB652` ("Demand Actual, NTS, D+6").
- Demand units: mscm converted to GWh with the documented factor 11.078.
- Weather: [Met Office HadCET daily mean temperature](https://hadleyserver.metoffice.gov.uk/hadobs/hadcet/data/download.html).

Run the complete operational cycle locally:

```bash
python scripts/update_data.py
python scripts/run_daily_forecast.py
python scripts/score_live_forecasts.py
python scripts/validate_oos.py
```

## Daily live forecasting and scoring

The GitHub Actions workflow runs at 14:30 and 17:30 UTC every day. The second run is an idempotent retry for a delayed D+1 publication. It does not depend on the Render VM being awake.

Each issue date gets one immutable forecast file under `reports/live_forecasts/`. The official target starts on the next UK gas day; the unknown issue day is bridged internally. A score is accepted only when the actual publication timestamp is later than the forecast issuance timestamp, preventing regenerated-history hindsight.

Scores are retained separately for provisional D+1 and finalized D+6 actuals. The report includes MAE, RMSE, bias, WAPE, interval coverage, per-horizon results, and skill versus persistence and weekly-naive baselines.

The workflow commits changed data, forecasts, and scores. Render must either have repository auto-deploy enabled or a deploy hook stored as the GitHub Actions secret `RENDER_DEPLOY_HOOK_URL`; the workflow supports both. The free Render service may sleep between requests, but an HTTP request wakes it and scheduled forecasting continues in GitHub Actions.

## Getting started

Requirements: Python 3.12 and pip.

```bash
pip install -r requirements.txt
python -m pytest tests/ -v
uvicorn api.app:app --reload
```

Run the API container:

```bash
docker build -t gas-forecast-api .
docker run --rm -p 8000:8000 gas-forecast-api
```

The Docker image serves FastAPI on port 8000 and includes the source data and generated validation report.

## API

Health and freshness:

```bash
curl http://127.0.0.1:8000/health
```

SARIMA forecast:

```bash
curl -X POST http://127.0.0.1:8000/forecast \
  -H "Content-Type: application/json" \
  -d '{"days": 7, "model_type": "sarima"}'
```

Weather-aware recursive forecast. One forecast temperature is required per day:

```bash
curl -X POST http://127.0.0.1:8000/forecast \
  -H "Content-Type: application/json" \
  -d '{"days": 3, "model_type": "linear", "mean_temperatures": [7.1, 6.8, 6.5]}'
```

Endpoints:

- `GET /health`: service freshness, D+1/D+6 cutoffs, and latest live-snapshot status.
- `POST /forecast`: on-demand dated forecasts using the D+6 history plus unresolved D+1 tail.
- `GET /live/forecast`: latest immutable scheduled forecast.
- `GET /live/performance`: frozen-forecast scores against D+1 and D+6 actuals.
- `GET /compare`: persisted rolling-origin validation report.
- `GET /docs`: interactive OpenAPI documentation.

Supported `model_type` values are `arima`, `sarima`, `linear`, and `random_forest`. Forecast horizons are capped at 31 days.

## Limitations

- CET is a national proxy rather than a demand-weighted spatial weather measure.
- Weather-aware validation uses realized temperature; operational accuracy depends on supplied weather forecasts.
- Calendar, holiday, price, storage, and structural-demand effects are not yet modeled.
- D+1 actuals are provisional operational signals; D+6 remains the finalized scoring truth.
- Live metrics are not meaningful until enough frozen forecasts have accumulated across dates and horizons.
- Forecast intervals are model-based for ARIMA/SARIMA and empirical training-residual intervals for the regression models; they are not yet formally calibrated.
- The fixed mscm-to-GWh factor is an analytical approximation rather than a daily calorific-value conversion.
